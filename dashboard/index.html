Gemma 4 Dropped 3 Days Ago. The Bugs Are Pouring In.

vLLM v0.19.0 shipped on April 1 with Gemma 4 support as the flagship feature. Three days later, Gemma 4 is the single biggest source of new issues. FP8 quantization produces gibberish, AWQ models hang with zero requests running, MoE variants crash on data parallelism, and tool calling is broken. If you're trying to serve Gemma 4 in production right now — you probably can't.

Why this matters: Gemma 4 was the headline feature of v0.19.0 and vLLM shipped a Docker image specifically for it (vllm/vllm-openai:gemma4). High-profile launch + broken day-3 experience = the kind of thing that erodes trust.
💥
Gemma 4 FP8 dynamic quantization = gibberish output (#39049)
FP8 quantization on Gemma 4 produces nonsensical output. Users expecting to run quantized Gemma 4 for cost savings are dead in the water.
🚫
Gemma 4 31B-AWQ: 0 running, 4 waiting — requests just hang (#39037)
AWQ-quantized Gemma 4 accepts requests but never processes them. 3 comments and counting.
🔧
Gemma 4 + tool calling is broken (#39043)
Tool/function calling doesn't work with Gemma 4. Given that tool use was a headline Gemma 4 feature, this is a big gap.

Other Issues Worth Watching

🔥 ROCm 0.19 Is Broken

⚠️
Hang during CUDA graph capture on ROCm in v0.19 (#39010)
6 comments — the hottest non-Gemma issue today. CUDA graph capture hangs on AMD GPUs. AMD users are stuck on v0.18.

⚡ Stability

💥
CUDA illegal memory access under high concurrency (Qwen3.5) (#39025)
CUDA graphs + high request volume = crash. The kind of thing that breaks prod on a Friday night.
🔌
DeepEP hardcodes quant param, causing MoE crash/accuracy loss (#39022)
DeepEP's all-to-all path forces per_act_token_quant=False. Models that need it on will crash or silently lose accuracy.

The Bottom Line

v0.19.0 is a rough release. The flagship feature (Gemma 4) has multiple blocking bugs across quantization, tool calling, and MoE. ROCm support regressed. Teams running production workloads should stay on v0.18.1 until the dust settles.

Qwen3.5 Is Having a Bad Week on New Hardware

The most popular MoE model is hitting problems on the newest chips. A 23% throughput regression on B200, garbled output on Intel XPU, and prefix caching issues. Meanwhile, Gemma 4 MoE can't even start with data parallelism, and the new expert load balancing infrastructure is deadlocking on Blackwell.

📈
23% throughput regression on Qwen3.5-397B NVFP4 decode (8xB200) (#39004)
5 comments. Flagship model on flagship hardware is 23% slower than last release. Likely a kernel config regression between v0.18 and v0.19.
📄
Qwen3.5 9B producing garbled output on Intel XPU (#38994)
Repetitive/garbled text on Intel backend. Correctness issues on non-NVIDIA platforms keep stacking up.
🔍
Qwen3.5 27B prefix caching issues (#38988)
3 comments. Prefix caching behavior is off for Qwen3.5 — a feature that matters for production latency.

MoE Infrastructure Growing Pains

💣
Gemma 4 MoE crashes with data parallelism > 1 (#38999)
The MoE variant (26B-A4B) can't do multi-GPU data parallel at all. AssertionError in the cache engine.
📦
EPLB rearrangement hangs with DP8 + EP on B200 (#38986)
Expert load balancing deadlocks on Blackwell. The new scaling infrastructure isn't ready for multi-GPU MoE yet.
💥
Gemma 4 MoE runtime MXFP4 quantization crashes during weight loading (#39000)
Online quantization + Gemma 4 MoE = crash during weight loading. Can't even get to inference.

Also Noted

🔌
Mistral Small 4 (119B MoE) fails to start on ROCm MI325X (#38972)
Two blocking issues preventing Mistral's big MoE from running on AMD's latest datacenter GPU. ROCm + MoE remains fragile.
🛠
v0.19.0 regression: page size not divisible by max page size (#38979)
Something broke between releases in the memory manager. A regression that didn't exist in v0.18.

The Bottom Line

New hardware + new models = pain. Qwen3.5 on B200 has a serious throughput regression. Gemma 4 MoE can't run multi-GPU. EPLB deadlocks on Blackwell. The pattern: vLLM's newest features (MoE infra, online quantization, Blackwell support) are hitting real-world edge cases that weren't caught before release.

v0.19.0 Day Two: 31 Issues. The Floodgates Are Open.

The busiest day of the week for vLLM issues. v0.19.0 dropped two days ago and users are hitting it hard. Qwen3.5 thinking mode is broken (14 comments — the hottest issue of the week), Gemma 4 tool parsing is mangling HTML, and Gemma 4 is painfully slow on older GPUs. Oh, and DeepSeek R1 output is wrong.

🔥
Qwen3.5 thinking mode: content goes to reasoning field, actual content = None (#38894)
14 comments. The hottest issue this week. When thinking is enabled, the model's response ends up in the wrong field and content returns empty. Completely broken for chain-of-thought.
💥
DeepSeek R1 produces incorrect output (#38931)
4 comments. Output correctness issue on one of the most popular models. Not gibberish — just wrong answers. Hard to debug, easy to miss.

Gemma 4: Day Two Issues

🔧
Gemma 4 tool parser duplicates HTML tag prefixes in streamed arguments (#38910)
4 comments. Streaming tool calls get corrupted HTML. Tool use + streaming is broken together — a common production pattern.
🐢
Gemma 4 extremely slow: ~9 tok/s on RTX 6000 with TRITON_ATTN fallback (#38887)
4 comments. On older/workstation GPUs, Gemma 4 falls back to Triton attention and crawls. 9 tokens/sec is unusable.
🚫
Gemma 4 31B freezes during loading on multiple RTX 6000 PRO (#38926)
3 comments. Can't even get to inference — hangs during weight loading on multi-GPU workstation setups.
🔌
Gemma 4 on Turing GPUs (SM 7.5): all attention backends hit shared memory limits (#38918)
3 comments. Gemma 4 can't run on Turing-generation GPUs at all. Shared memory requirements exceed what the hardware provides.
🛠
#38946 Gemma 4 tool usage produces invalid JSON in streaming CLOSED
Fixed quickly — streaming tool calls produced malformed JSON. One of the few Gemma 4 bugs with fast turnaround.

Also Noted

💻
H100 CUDA 13.0 nightly wheels require glibc 2.35, failing on 2.34 (#38908)
3 comments. Docker images built against a newer glibc than many production hosts run. Blocks upgrades.
🕑
Qwen3.5 inference TimeoutError with FlashInfer GDN backend (#38916)
Qwen3.5 times out on the FlashInfer backend. Another backend-specific failure for the most popular MoE model.

The Bottom Line

Post-release flood day. 31 issues — nearly 3x the daily average. The Qwen3.5 thinking bug (14 comments) is the week's hottest issue. Gemma 4 problems are spreading beyond quantization into tool parsing, performance on non-datacenter GPUs, and basic model loading. The pattern: v0.19.0 shipped ambitious features but QA didn't cover the combinatorial matrix of models x hardware x features.

v0.19.0 Day One: The Early Reports Are In

First full day after the v0.19.0 release. 21 issues filed — above average but not yet a flood. The first Gemma 4 reasoning parser bug surfaces, some vLLM IR infrastructure work lands, and users are kicking the tires on new features like ASR models and PDL support.

💬
Gemma 4 reasoning parser fails — channel tokens stripped incorrectly (#38855)
5 comments. The first Gemma 4 bug to surface post-release. Reasoning parser can't separate thinking content from response content. Early signal of what becomes a theme this week.
🔍
Port configuration question (#38820)
6 comments. A usage question that got a lot of engagement — people are actively trying to deploy v0.19.0 and hitting basic setup friction.
📹
Qwen3-VL video inference: metadata errors (#38811)
3 comments. Video inference on Qwen3-VL is broken due to missing metadata handling. Multimodal edge cases continue to bite.

Feature Requests & Infrastructure

🚀
Feature: General LL GEMMs with PDL support (#38772)
3 comments. Request for Programmatic Dependent Launch support in low-latency GEMMs. Forward-looking GPU kernel work.
🔌
Feature: ROCm support for Kimi K2.5 EAGLE3 MTP heads (#38851)
Request to bring speculative decoding (Eagle3) for Kimi K2.5 to AMD GPUs. Shows growing AMD user demand for advanced features.
🛠
vLLM IR: Op test & benchmark infra (#38782)
vLLM IR (intermediate representation) is a new compiler infrastructure project. Early-stage but potentially big — could change how vLLM optimizes model execution.
🔌
Does vLLM support online inference for Qwen3 ASR forced aligner? (#38805)
Users already trying the new audio model support. Online inference for ASR models apparently not yet wired up.

The Bottom Line

Day one after release — early adopters are exploring. Issue volume is elevated but not alarming yet. The Gemma 4 reasoning parser bug is the first canary. Feature requests for ROCm spec decode and vLLM IR infrastructure show the community is engaged and forward-looking. The real flood comes tomorrow.
