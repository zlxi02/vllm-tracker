Gemma 4 Launch Week Gets Messy: Tool Calling Broken, Thinking Mode Fails

Today in vLLM: Five days after Gemma 4 shipped in v0.19.0, the model is generating a steady stream of critical bugs. Tool calling is fundamentally broken, thinking mode parsing fails completely, and structured outputs are producing garbage.

Why this matters: Why this matters: Gemma 4 was the flagship feature of v0.19.0, but core functionality is unusable. This suggests insufficient testing of the new model architecture before release.
🔥
Gemma4 tool-call-parser corrupts boolean values (#39089)
Tool calling parser mangles boolean arguments, making function calls unusable. Critical for production deployments.
💥
Gemma4 tool calling validation failures with PI coding agent (#39072)
Required properties missing from tool calls when using Gemma4 with coding agents. Shows broader tool integration issues.
💩
Gemma4 31B structured outputs producing weird characters (#39071)
Structured output generation returns garbled characters instead of proper JSON. User suspects 'quick solve' available.
🤡
Gemma4 reasoning parser fails completely — thinking tokens stripped before parsing (#38855)
Channel tokens get stripped before reaching the reasoning parser, causing all thinking content to leak into regular content.
👻
Gemma4 tool parser duplicates HTML tag prefixes in streaming (#38910)
Streaming tool arguments get corrupted with duplicate HTML prefixes. Affects real-time tool calling workflows.

Qwen3.5 Performance & Memory Issues

💥
Qwen3.5 warmup kernels leak 3.4 GiB GPU memory despite empty_cache() (#36973)
Triton autotuned GDN kernel warmup permanently consumes GPU memory. 23 comments suggest this is a widespread issue.
📉
23% output throughput regression on Qwen3.5-397B NVFP4 decode over 10 days (#39004)
Systematic performance degradation from 47,200 to 38,300 tok/s on 8×B200. Three distinct regression points identified.
💥
CUDA illegal memory access with CUDA graphs under high concurrency (#39025)
Qwen3.5-35B-A3B crashes with memory access violations when using CUDA graphs with tensor parallelism.

Infrastructure & Compatibility

⚠
Gemma4 hits shared memory limits on Turing GPUs (SM 7.5) (#38918)
All attention backends fail on older Turing architecture due to memory constraints. Affects RTX 20 series users.
🔧
H100 CUDA 13.0 nightly wheels require glibc 2.35, fail on 2.34 systems (#38908)
Distribution compatibility regression breaks deployments on slightly older Linux systems.
🔥
vLLM 0.19 loses tool calls when XML emitted inside reasoning blocks (#39056)
Tool calling breaks when models emit XML markup inside thinking regions. Affects reasoning-enabled workflows.

Closed Issues & Fixes

✅
Qwen3.5 thinking mode outputs reasoning in content field CLOSED (#38894)
Reasoning parser fix for content field contamination. Shows some progress on parser reliability.
✅
Gemma4-31B-AWQ stuck in waiting state CLOSED (#39037)
AWQ quantization scheduling issue resolved. At least some Gemma4 configurations are working.

The Bottom Line

v0.19.0 is shaping up to be a problematic release. Gemma 4's core features are broken, Qwen3.5 has serious memory leaks and performance regressions, and infrastructure compatibility has regressed. Teams should stay on v0.18.1 until these issues are resolved.
